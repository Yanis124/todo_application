# ADR-001: Technology Stack and Architecture Decisions for To-Do App

## Status
Accepted

## Date
2025-01-27

## Context
We need to build a To-Do application with the following requirements:
- User authentication via JWT
- Full CRUD operations on tasks (title, description, priority, status, due date)
- A filterable dashboard
- REST API with Swagger documentation
- Stack mandated: FastAPI + SQLite + React

## Decisions

### D1: Backend Framework — FastAPI
**Decision:** Use FastAPI as the backend framework.

**Rationale:**
- Mandated by project specs
- Built-in OpenAPI/Swagger documentation at `/docs`
- Native async support for high performance
- Pydantic models for request/response validation
- Dependency injection system ideal for auth middleware

**Consequences:**
- Python 3.9+ required
- Team must be familiar with async Python patterns
- Automatic API documentation reduces documentation effort

### D2: Database — SQLite with SQLAlchemy ORM
**Decision:** Use SQLite as the database, accessed via SQLAlchemy ORM.

**Rationale:**
- SQLite mandated by project specs
- Zero-configuration, file-based database — ideal for single-server deployment
- SQLAlchemy provides ORM layer for cleaner code and easier migrations
- Alembic can be used for schema migrations if needed

**Consequences:**
- No concurrent write scaling (acceptable for a to-do app)
- Database is a single file — easy backup but no remote DB server
- SQLAlchemy adds a dependency but provides migration path to PostgreSQL if needed later

**Alternatives considered:**
- Raw SQL with `sqlite3` module — rejected for lack of ORM benefits and migration support
- Tortoise ORM — rejected as SQLAlchemy is more mature and widely adopted

### D3: Frontend — React (Single Page Application)
**Decision:** Use React to build a single-page application (SPA).

**Rationale:**
- Mandated by project specs
- Component-based architecture suits the dashboard + forms pattern
- Rich ecosystem for state management and HTTP clients

**Consequences:**
- Separate frontend build process (Node.js/npm required for development)
- Need CORS configuration on FastAPI backend
- API-first architecture enables future mobile clients

**Sub-decisions:**
- Use `axios` or `fetch` for HTTP calls to the API
- Use React Router for client-side routing (login, register, dashboard)
- Use functional components with hooks (modern React patterns)

### D4: Authentication — JWT with Bearer Token
**Decision:** Implement JWT-based authentication using `python-jose` and `passlib[bcrypt]`.

**Rationale:**
- JWT mandated by project specs
- Stateless authentication — no server-side session storage needed
- Standard Bearer token scheme works well with SPA frontends

**Consequences:**
- Tokens must be stored client-side (localStorage or httpOnly cookies)
- Token expiry must be managed (access token + optional refresh token)
- Secret key must be configured via environment variable

**Security considerations:**
- Passwords hashed with bcrypt (via passlib)
- JWT secret key stored in environment variables, not in code
- Token expiry set to 30 minutes by default
- HTTPS recommended for production

### D5: API Design — RESTful with OpenAPI
**Decision:** Follow REST conventions with automatic OpenAPI/Swagger documentation.

**Rationale:**
- REST API mandated by project specs
- FastAPI auto-generates Swagger UI at `/docs`
- Pydantic schemas ensure request/response validation and documentation

**API Structure:**
```
POST   /api/auth/register    — User registration
POST   /api/auth/login        — User login (returns JWT)
GET    /api/tasks             — List tasks (with filters)
POST   /api/tasks             — Create task
GET    /api/tasks/{id}        — Get task detail
PUT    /api/tasks/{id}        — Update task
DELETE /api/tasks/{id}        — Delete task
```

### D6: Project Structure
**Decision:** Monorepo with separate `backend/` and `frontend/` directories.

```
/
├── backend/
│   ├── app/

│   │   ├── database.py       — SQLAlchemy engine & session
│   │   ├── models/           — SQLAlchemy models
│   │   │   ├── user.py
│   │   │   └── task.py
│   │   ├── schemas/          — Pydantic schemas
│   │   │   ├── user.py
│   │   │   └── task.py
│   │   ├── routers/          — API route handlers
│   │   │   ├── auth.py
│   │   │   └── tasks.py
│   │   ├── services/         — Business logic
│   │   │   ├── auth_service.py
│   │   │   └── task_service.py
│   │   └── dependencies.py   — Auth dependency (get_current_user)

│   └── alembic/ (optional)
├── frontend/
│   ├── src/
│   │   ├── components/
```

- Clear separation of concerns
- Independent deployment possible
- Shared repository simplifies development workflow

### D7: Task Data Model
**Decision:** Task entity with the following fields:

| Field       | Type     | Required | Notes                          |
|-------------|----------|----------|--------------------------------|
| id          | Integer  | Auto     | Primary key                    |
| title       | String   | Yes      | Max 200 chars                  |
| description | Text     | No       | Optional long text             |
| priority    | Enum     | Yes      | low, medium, high              |
| status      | Enum     | Yes      | todo, in_progress, done        |
| due_date    | DateTime | No       | ISO 8601 format                |
| created_at  | DateTime | Auto     | Set on creation                |
| updated_at  | DateTime | Auto     | Updated on modification        |
| user_id     | Integer  | Auto     | Foreign key to User, set by JWT|

**Rationale:** Directly maps to spec requirements. Enums enforce valid values.

### D8: User Data Model
**Decision:** User entity with the following fields:

| Field         | Type     | Required | Notes                    |
|---------------|----------|----------|--------------------------|
| id            | Integer  | Auto     | Primary key              |
| email         | String   | Yes      | Unique, used for login   |
| hashed_password | String | Yes      | bcrypt hash              |
| created_at    | DateTime | Auto     | Set on creation          |

## Risks

1. **SQLite concurrency:** SQLite has limited write concurrency. Acceptable for a to-do app but would need migration to PostgreSQL for high-traffic scenarios.
2. **JWT in localStorage:** Vulnerable to XSS attacks. Mitigation: sanitize all inputs, consider httpOnly cookies for production.
3. **No email verification:** Registration doesn't verify email. Acceptable for MVP scope.
